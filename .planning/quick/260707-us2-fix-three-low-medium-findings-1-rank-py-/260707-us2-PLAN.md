<objective>
Fix three Low/Medium findings from an independent Antigravity/agy code audit:

1. **rank.py timezone risk (Medium)**: `creation_velocity` parses `created_at_iso` and
   `captured_at_iso` via `datetime.fromisoformat`. `captured_at_iso` is always tz-aware
   (produced via `datetime.now(timezone.utc)`), but `created_at_iso` (from PyGithub's
   `repo.created_at.isoformat()`) could theoretically be naive on an untested PyGithub
   version/path — subtracting a naive datetime from an aware one raises `TypeError` and
   crashes the whole ranking pipeline. Fix: if a parsed datetime has `tzinfo is None`,
   treat it as UTC.

2. **daily.yml UTC-midnight race (Low)**: the collector step and the email step each
   independently compute "today's" date. The collector runs for several minutes
   (search + refresh + write); if that run straddles UTC midnight, the email step's
   fresh `datetime.now(timezone.utc)` call disagrees with the date the collector
   actually used, finds no matching report file, and silently skips sending — no error,
   just a missing email. Fix: capture the collector's *actual* report file path
   (globbed immediately after the collector step, before any other steps run) and pass
   it forward via `$GITHUB_OUTPUT`; the email step reads that path instead of
   recomputing a date. `write_digest` runs unconditionally on every successful collector
   run (verified in `src/collector.py` step 5 — no conditional skip), so the freshly
   written report is always the newest file in `reports/` by the time this step runs.

3. **daily.yml hardcoded recipient (Low)**: the email step hardcodes
   `midnyter@gmail.com` directly in the workflow YAML. Move it to a repository variable
   (`vars.REPORT_TO_EMAIL` — not a secret, an email address isn't sensitive) referenced
   via `${{ vars.REPORT_TO_EMAIL }}`.

Purpose: close three confirmed audit findings — one crash risk, one silent-failure risk,
one hardcoded-config smell — without touching unrelated code paths.

Output: `src/rank.py` tzinfo guard + regression test; `.github/workflows/daily.yml`
race-free report-path passing + repo-variable recipient.
</objective>

src/rank.py (current, lines 33-51):
```python
def creation_velocity(stars: int, created_at_iso: str, captured_at_iso: str) -> float:
    created = datetime.fromisoformat(created_at_iso)
    captured = datetime.fromisoformat(captured_at_iso)
    age_hours = (captured - created).total_seconds() / 3600
    age_hours = max(age_hours, config.AGE_HOURS_FLOOR)  # floor: avoid div-by-zero
    return (stars / age_hours) * 24  # → stars/day
```
`from datetime import date, datetime, timedelta, timezone` is already imported at module
top — `timezone` is already available, no new import needed.

Sibling functions (`spike_velocity`, `rolling_velocity`, `is_new`) are NOT touched by this
plan: `spike_velocity`/`rolling_velocity` only ever subtract two `captured_at` values,
both always tz-aware from the same snapshot-writer source; `is_new` only calls `.date()`
on one parsed datetime, never subtracts two. `creation_velocity` is the only site that
subtracts a metadata-sourced (possibly naive) datetime from a snapshot-sourced (always
aware) one — the genuine root cause, not just the symptom the audit named.
