# Phase quick-260701-u16: Port CATEGORY LEADERS grid + stats strip to match 4a reference Summary

**Full pixel port of the CATEGORY LEADERS grid and footer stats strip from the original "4a" Claude Design handoff, replacing the blunt min-height patch from 260701-tnx. Ported the reference's bottom-anchored card structure, compact sizing, joined stats strip, and NEW-dot indicator into Gmail-safe table markup (no flex/grid/gap). Also fixed a real digit-dependent baseline-wobble bug discovered during visual verification (headless-Chrome screenshot diffing).**

## Accomplishments
- `_render_leader_cell`: kicker (9px, was 9.5px) pinned flush top and body pinned flush bottom via a nested `<table height="46">` with `valign="top"`/`valign="bottom"` rows — Gmail-safe equivalent of the reference's `display:flex; justify-content:space-between`. Card `border-radius:7px`/`padding:9px 10px`/`min-height:64px` (was 10px/14px 16px/96px).
- Active-leader body: value (17px Newsreader) + `★/d` unit on one line via a single-row 2-cell `<table valign="middle">` (not inline spans — see bug fix below); repo name row now shows a small green NEW dot when the leader entry counts as new (new requirement, ported from the reference, using the existing `markers.get(str(id), "new") == "new"` rule).
- Empty state: `font-size:12px`, `line-height:1.3`, `margin-top:5px` (was 14px/10px), still no number/unit text.
- Section label: "Category leaders" gray `#5b6573` 9px mono (was green 11px "Category Leaders").
- `_render_strip_cell`/stats strip: rewritten as ONE joined `<table>` with `border`/`border-radius:7px`/`background:#0e0f12` on the outer table, `border-right` dividers between cells, value+label on the same baseline line — replacing three separately-bordered/rounded cards.
- **Bug fix during visual QA**: user flagged the "brand new week" card's `★/d` unit looked vertically off relative to its number. Root cause: serif oldstyle/non-lining numerals (digit-dependent descenders) made inline baseline-alignment wobble based on which digits were present. Verified via headless-Chrome screenshot crop comparison (94.9 vs 3692.2 side by side) before/after. `font-variant-numeric:lining-nums` had no visible effect; fixed by switching value+unit to a `valign="middle"` table-cell pair, confirmed fixed via a second screenshot comparison.
