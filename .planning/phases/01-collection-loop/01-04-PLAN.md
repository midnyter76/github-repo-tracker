<objective>
Wire the loop together and ship it to GitHub Actions: `collector.py` builds the
token-authenticated client (DATA-01, AUTO-02), orchestrates discovery + re-fetch +
persistence, and `daily.yml` runs it on a daily UTC cron with secret injection and
commit-back (AUTO-01, AUTO-02, AUTO-03). This is the plan that makes "a snapshot
appears in the repo every morning" true.

Purpose: closes the fetch→snapshot→commit loop and is the phase's primary security
surface — the GITHUB_TOKEN must be injected from a secret and never echoed, logged,
or committed.
Output: `src/collector.py`, `tests/test_collector.py`, and `.github/workflows/daily.yml`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions secret store → collector process env | `secrets.GITHUB_TOKEN` is injected as an env var for the run step only |
| collector process → committed repo (commit-back) | git-auto-commit pushes data/** using the workflow's GITHUB_TOKEN credentials |
| committed workflow + logs → public repo | The repo is public (D-08); logs and committed files must contain no secret |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-10 | Information Disclosure | token in Actions logs | mitigate | Token referenced ONLY as `env: GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}`; never in a `run:` echo or base64; `build_client` reads it once and never prints it (Pitfall 4). Grep gates in both tasks enforce this. |
| T-01-11 | Information Disclosure | token committed to repo | mitigate | Token lives only in `os.environ` for the process lifetime; no `.env` file exists; commit-back uses `file_pattern: "data/**"` so only data files are committed — never source/env. |
| T-01-12 | Elevation of Privilege | over-broad GITHUB_TOKEN scope | mitigate | `permissions: contents: write` is the minimal scope the auto-commit action needs; no `packages`, `id-token`, or `actions` write granted (RESEARCH §Security Domain). |
| T-01-13 | Denial of Service | commit-back self-triggering the workflow (infinite loop) | mitigate | `[skip ci]` in the commit message plus GitHub's built-in rule that GITHUB_TOKEN pushes do not re-trigger workflows (RESEARCH Pattern 8 note) — belt and suspenders. |
| T-01-14 | Tampering | malicious dependency executing at install | mitigate | `uv sync` installs from the committed, pinned `uv.lock` (Plan 01); no floating versions resolved at CI time. |
</threat_model>
