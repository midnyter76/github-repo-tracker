<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| collector.run() orchestration | Sequences untrusted-data flow (snapshots/metadata -> buckets -> public digest) and the seen-store write. A wrong write-order can mark repos seen without ever publishing them. |
| GitHub Actions token | GITHUB_TOKEN (Phase 1) authorizes the commit-back of reports/**; the token is injected from a secret and never echoed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Repudiation / data loss | seen saved before report | mitigate | run() writes the digest BEFORE save_seen (D-10); a crash between steps leaves the repo flagged 🆕 next run rather than silently-seen. Asserted by ordering test. |
| T-02-11 | Information Disclosure | token in logs / digest | mitigate | No new os.environ references (invariant grep == 1); token stays Phase-1 build_client only; never written to reports/. |
| T-02-12 | Tampering | uncommitted digest (Pitfall 6) | mitigate | daily.yml file_pattern extended to `reports/**` so the digest is actually committed (REPORT-01 in production), no behavior depends on unverified manual steps. |

## Trust Boundaries — note
